# 技术方案设计协议 (Technical Design Protocol)

## 🛑 Phase 0: 核心设计哲学
**决策优先级：可用性 > 简洁性 (KISS) > 可扩展性 (OCP)**

1. **YAGNI 原则**：除非用户明确要求，否则不要为"未来"写代码
2. **KISS 原则**：即使符合设计模式，如果代码难以阅读，也是失败的设计
3. **Safety First**：任何输入输出都必须考虑边界情况和安全性

### 使用场景
- **必须使用**：新增适配器、核心架构变更、复杂功能模块（预计 > 500 行代码）
- **建议使用**：新功能开发涉及多个模块交互、性能优化方案
- **不需要使用**：简单 Bug 修复、添加辅助函数、配置项调整（参考 `/prompts/bug-fixing.md` 和 `/prompts/feature-development.md`）

## 🔄 Phase 1: 深度分析与盘问 (Context Analysis)

**🛑 STOP! 不要立即生成解决方案！** 首先分析相关需求文档和代码，然后执行以下步骤：

1. **资产扫描**：
   - 检查现有的 `models/`、`adapters/`、`utils/` 模块
   - 查找可复用的代码片段和类型定义
   - 识别现有的事件模式和配置结构

2. **约束识别**：
   - 技术栈：Python 3.8+、asyncio、Web3.py、Pydantic
   - 架构约束：适配器模式、事件驱动、异步优先
   - 性能要求：高吞吐量、低延迟

3. **澄清问题**：
   - 列出 3-5 个关键问题
   - 特别关注区块链相关的边缘情况（网络中断、分叉、重组等）
   - 明确错误处理预期

**（输出上述内容并等待用户确认，再进入 Phase 2）**

## 📝 Phase 2: 技术方案草案 (Drafting the Tech Spec)

收到用户回复后，生成技术方案文档（保存为 markdown 文件）：

* **架构设计**：
  - 新组件如何融入现有架构
  - 适配器或事件处理器的交互关系
  - 复杂交互提供 Mermaid 流程图

* **接口定义**：
  - 类的公共方法签名
  - 事件类型定义
  - 配置参数说明

* **数据模型**：
  - Pydantic 模型定义
  - 事件数据结构
  - 配置结构变更

* **错误处理**：
  - 区块链特定异常（网络错误、交易失败等）
  - 超时和重试机制
  - 错误恢复策略

* **关键限制**：
  - 对于复杂逻辑，使用自然语言描述
  - （非常重要）避免直接代码实现，保持设计层面

## ⚖️ Phase 3: 对抗性审计 (Adversarial Audit)
**切换角色：现在你是一位“挑剔的资深架构师”和“安全专家”。**
请对 Phase 2 的草案进行攻击性审查，而非简单的确认。

**审查清单 (Pass/Fail):**

1. **[Architecture] 架构解耦**:
   - Check: 新组件是否依赖具体的区块链实现而非接口？
   - Check: 修改某个适配器是否会影响其他适配器？
   - Check: 事件处理逻辑是否与链特定细节耦合？

2. **[Security] 安全性**:
   - Check: 私钥或敏感信息是否会意外暴露？
   - Check: 区块链地址、交易哈希等输入是否经过验证？
   - Check: 是否存在重放攻击或篡改风险？

3. **[Reliability] 可靠性**:
   - Check: 网络中断时是否有适当的重连机制？
   - Check: 区块链重组时的事件处理策略？
   - Check: 长时间运行是否存在内存泄漏？

4. **[Performance] 性能考量**:
   - Check: 事件处理是否会阻塞主循环？
   - Check: 批量处理 vs 单条处理的权衡？
   - Check: WebSocket 连接数是否合理？

5. **[KISS] 过度设计**:
   - Check: 是否引入了不必要的抽象层？
   - Check: 能否用更简单的组合代替复杂模式？

6. **[Blockchain] 区块链特定**:
   - Check: Gas 费用、区块确认数等参数是否可配置？
   - Check: 是否考虑了不同链的差异（如 Solana vs Ethereum）？

**输出要求**:
* 如果所有项 Pass，输出 "✅ Architecture Approved"。
* 如果有 Fail，输出 **“⚠️ Refactoring Required”**，并给出具体的修正后的代码片段或逻辑调整。

**修改方案**:
* 如果有 Fail，询问用户是否针对此输出的Fail对方案文档进行修改，如是请进行必要的，最小程度的技术方案修改。

## ✅ Phase 4: 开发执行计划 (Implementation Plan)

将方案转化为**原子化**的开发步骤（每步代码变更是可独立 commit 的）：

1. **[ ] Step 1 (Models)**: 定义 Pydantic 模型和事件类型
   - 创建新的事件模型类
   - 更新配置结构（如需要）
   - 编写模型验证测试

2. **[ ] Step 2 (Adapter/Core)**: 实现核心适配器或处理器
   - 继承 BaseAdapter 实现新适配器（如适用）
   - 实现事件处理逻辑
   - 编写单元测试（TDD）

3. **[ ] Step 3 (Integration)**: 集成到 ChainListener
   - 注册新适配器
   - 更新配置解析逻辑
   - 实现事件回调机制

4. **[ ] Step 4 (Tests)**: 完善测试覆盖
   - 集成测试
   - 错误场景测试
   - 性能基准测试

5. **[ ] Step 5 (Documentation)**: 更新文档
   - API 文档
   - 使用示例
   - README 更新（如需要）

## 技术方案输出要求
* **文档结构**: 文档需要有清晰，层次合理的章节结构。
* **输出内容控制**: 不要输出与技术方案无关的内容。
